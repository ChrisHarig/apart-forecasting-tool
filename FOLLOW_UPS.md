## 1. Predictions feature — local + HF (multi-phase)

Goal: users can upload prediction CSVs locally, compare them to the
EPI-Eval truth datasets, and (eventually) submit them to a per-target
sibling dataset on HuggingFace for review and inclusion in a public
leaderboard.

### Locked-in design decisions

- **Storage on HF: sibling-dataset pattern.** Each target dataset gets
  an `EPI-Eval/<id>-predictions` companion (e.g. `nhsn-hrd-predictions`)
  that accumulates rows from every submitter. Subfolder pattern would be
  nicer but HF's viewer support for nested folders is patchy.
- **Quantile representation: long format.** One row per
  `(target_date, [categorical dims…], target_column, quantile, value)`.
  Quantile is a float in [0,1]; **point estimates use `quantile=NULL`**
  (one canonical row per point estimate, no sentinel strings). Flexible
  to any quantile set, clean WIS computation.
- **Join contract: `date` + every non-numeric column in the prediction
  CSV.** No row-number matching. For categorical dims the target truth
  has but the CSV lacks (e.g. `(date, value)` uploaded against a
  multi-state truth), prompt the user once per missing dim at upload
  (dropdown of distinct truth values). Length is a sanity-check warning
  ("CSV has 12 rows, joined truth slice has 11"), not the join key.
  Generalize as much as we can, then add labeling toggles (group-by
  set, filtered subset) if it gets awkward in practice.
- **Compare-to join type: outer.** Predictions for dates past
  `time_coverage.end` show on the chart and table; comparison metrics
  compute only on the overlap.
- **Date validation: hard-reject at upload** any `target_date` outside
  `[truth.start, truth.time_coverage.end + ~2 months]`. Forecast-horizon
  cap matches the v2 prediction time-range rule.
- **rWIS divide-by-zero: flag the row.** When the baseline WIS is 0
  (e.g. naive-last-value on an already-observed date), surface the
  affected rows in the metrics panel rather than dropping or capping.
- **WIS reference: Bracher et al. 2021** for α-weighting and `K+0.5`
  normalization. Test fixtures cite the paper so the convention is
  unambiguous.
- **Verification: HF community PR review.** Submitter clicks "Open PR
  to HF" in our dashboard → we use the HF API to create a community PR
  on the sibling dataset → maintainer reviews + merges via HF's web
  PR UI. Not auto-merge: a human eyes every submission.
- **Map display: replace mode** (defer to v2 / v3). For now, predictions
  show on the chart and table only.
- **Baselines: at least naive-last-value.** Plus naive-last-week,
  seasonal-naive, and linear-trend if cheap. ARIMA / SARIMA / TabPFN
  later.
- **rWIS baseline default: naive-last-value** — simplest possible
  comparison, user can swap to a different baseline.

### v1 — local-only, chart + table (no map, no HF push)

- Drag-and-drop CSV upload UI in the dashboard. "My datasets" section
  alongside the EPI-Eval catalog.
- Parse + validation:
  - Hard: `date` column parseable + within the target's acceptable
    range (per locked-in date-validation rule).
  - Optional categorical columns; user prompted for any truth
    categorical dim missing from the CSV (one dropdown per dim).
  - Optional `quantile` column; values in [0,1] with `NULL` reserved
    for the point-estimate row.
  - At least one numeric value column.
- Storage: in-memory for the session (drag-drop again to bring back
  after reload). IndexedDB persistence is a v1.5 add-on.
- Compare-to picker: select an EPI-Eval dataset + target column. Outer
  join on `date` + the prediction's categorical dims; overlay on the
  time-series chart and the seasonal chart. Predictions show in the
  data table alongside truth, including for dates past truth coverage
  (metrics compute only on the overlap).
- Metrics panel:
  - MAE, RMSE, MAPE, signed bias (point predictions)
  - Per-quantile coverage (for the 50%, 80%, 95% intervals if quantile
    rows are present)
  - WIS (Weighted Interval Score, from quantile rows)
  - rWIS relative to naive-last-value (or whichever baseline picked)
- Built-in baselines, generated on demand via a "Run baselines" button:
  - **naive-last-value** — predicted = most recent observed
  - **naive-last-week** — predicted = value at `target_date - 7 days`
  - **seasonal-naive** — predicted = value at `target_date - 1 year`
    (week-of-year matched)
  - **linear-trend** — linear fit on last N=8 weeks, extrapolate
- Sample dataset checked into `public/examples/`: a synthetic flu
  forecast targeting `nhsn-hrd.totalconfflunewadm` for California
  (FIPS 06). Generated by perturbing real `nhsn-hrd` values + emitting
  realistic quantile bands. ~260 rows, ~5 KB. Demos the whole loop —
  drag-drop, see WIS / coverage / baseline-comparison.
- Tests: pure-function tests for the metrics module (MAE / RMSE /
  bias / WIS / rWIS / coverage), the CSV parser, the baseline
  generators. The math is the easy-to-screw-up part — thorough
  coverage there is the highest-leverage testing in this feature.

Effort: 2–3 sessions.

### v2 — HF predictions sibling datasets + community PR submission

- Schema for prediction cards as `<id>-predictions` siblings, plus
  schema doc updates.
- Validator extension in `upload_pipeline/core/validate.py` that knows
  the prediction schema (accepts `target_dataset` + `target_column`,
  validates `quantile ∈ [0,1]`, checks the target dataset/column exists).
- Required fields for HF-published predictions: `submitter`,
  `model_name`, `description` (human-readable), `target_dataset`,
  `target_column(s)`, dates.
- "Open PR to HF" button in the dashboard. Uses HF API
  (`huggingface_hub` JS bindings or direct REST) to:
  1. Fork the sibling dataset to the user's account
  2. Add their parquet under `data/<submitter>-<timestamp>.parquet`
  3. Create a community PR back to `EPI-Eval/<id>-predictions`
- Pull-predictions toggle when opening a target dataset — fetches the
  sibling dataset's slice and shows accepted predictions overlaid on
  the chart and listed in the table.
- Prediction time range: anywhere between the start of the target
  dataset's coverage and ~2 months past its `time_coverage.end`.
  The dashboard simply doesn't show comparison metrics for predictions
  whose target_date is past the truth's coverage.

Effort: 2–4 sessions, mostly HF API plumbing + schema work.

### v3 — map display + advanced baselines

- **Map (replace mode)**: when a prediction set is active, the
  chloropleth paints predicted values instead of observed. The time
  scrubber can move past `time_coverage.end` into the forecast horizon.
- **Diff mode** (alternative): paint `prediction - truth` residuals so
  the user sees *where* the model over/under-shoots.
- **ARIMA / SARIMA baselines** — via a JS library
  (e.g. `arima` npm package) or WASM port. Real engineering since these
  are iterative fits, but cheap relative to TabPFN.
- **TabPFN baseline** — needs WASM port or a server-side runner. Big
  lift; defer until #2's HF submission flow has settled.

Effort: 1–2 sessions for the map work, separately ~1 week for ARIMA/
SARIMA, and TabPFN is its own multi-week project.

## 2. Boundary loading — cold-load perf (continuation)

Already shipped: timing logs, per-country-set filter cache, error
surfacing through the renderer, and a real spinner. If the *cold*
load is still slow on the iso3166-2 admin-1 file (2.2 MB raw, 757 KB
gzipped), the next moves are:

1. Convert the admin-1 GeoJSON → TopoJSON (~3-5× smaller on disk; gzip
   makes it tiny on the wire). Same pattern as `usAtlas`. Vendor the
   converted file via mapshaper or topojson CLI.
2. Move the JSON parse off the main thread (Web Worker). Bigger lift,
   only worth it if (1) isn't enough.

## 3. Time-series chart — downsampling toggle

The time-series chart plots whatever cadence the data is in. Daily
sources show daily points (5,000+ on a 5-year window). A
"display at: daily / weekly / monthly / yearly" toggle would aggregate
on the fly using the column's declared aggregation and dramatically
reduce visual noise. Composes naturally with the existing Metric /
Split by / Filters controls.

The seasonal chart already does period-bucketing — this is the
*other* axis (timeline view, not period overlay).

## 4. Seasonal chart — deferred polish

v1 shipped. Items deferred:

- **Per-source `default_period` in card metadata.** So e.g. NHSN HRD
  defaults to flu-season-northern instead of calendar-year. New
  optional field in `card.yaml`; reader picks it up; UI uses as default.
- **Split-by in seasonal mode.** Currently shows a small "Split-by
  ignored in seasonal mode" badge and proceeds without it. Real story:
  N split-keys × M periods → potentially explosive line count. Wire
  through with a warning when the product exceeds ~20.
- **Partial-period annotation.** Lines that stop mid-period (the
  in-progress current period, or pre-coverage years) get a small
  "thru-W14 / partial" badge so the user knows it's incomplete, not
  abnormally low.
- **Period-over-period Δ overlay.** Each line shows the difference
  from the prior period. Useful but a separate mode toggle.
- **Custom-period definition UI.** Power-user feature; let users
  define "summer 2024" or arbitrary windows.
